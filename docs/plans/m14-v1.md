# M14 — v1.0: the proof and the polish

**Blocked on decisions:** none new — this milestone *closes* decisions.
Depends on everything (M3–M13).
**Error codes:** none new; audit instead.

## Goal

Prove the language does what the README will claim, fix what the proof
exposes, and cut v1.0. The thesis to validate: *a competent programmer
can rewrite a small real-world Rust/Go/C tool in Lex with less friction
and no loss of safety or meaningful performance.*

## Workstream 1 — The rewrite proof

Build THREE real tools in `showcase/`, each a known quantity with a
reference implementation to compare against:

1. **`lexgrep`** — grep-lite: file walking, string search, CLI flags,
   exit codes. Exercises fs/io/args/errors/slices.
2. **`jsonfmt`** — JSON pretty-printer/validator with good error
   positions. Exercises std/json, enums, recursion, Result plumbing.
3. **`wordfreq-par`** — parallel word frequency across many files.
   Exercises tasks/channels/Map/sort_by/closures.

For each: a `NOTES.md` recording (a) every place Lex fought the author —
these become issues, (b) line count vs the reference, (c) `hyperfine`
benchmark vs the reference build — target ≤1.5× the Rust reference
runtime (transpiled output should be ~1×; investigate anything worse,
it's usually an accidental clone — make L0201/L0501 catch it).
Showcase tools are golden-tested like examples (fixed inputs, pinned
outputs) and become permanent regression armor.

## Workstream 2 — Audits (each produces fixes, then a checklist sign-off)

- **Diagnostics audit:** read every snapshot in tests/ui/ end to end
  against docs/04 voice rules; fix drift; verify every E-code has
  what/why/fix and a `.fixed.lex` where applicable; kill dead codes.
- **Soundness audit:** fuzz the gap between sema and rustc — a script
  generates random well-typed-per-sema programs (grammar-directed, seed
  corpus = all examples) and asserts rustc accepts them; run a long
  soak locally, a short one in CI. Every divergence is a P0 (I2/R2).
- **Performance audit:** compile-time budget — `lex build` on the
  showcase tools; profile front end vs rustc share; cache generated
  .rs + binary by content hash in ~/.cache/lex so unchanged rebuilds
  are instant (`lex run` repeat <100ms).
- **API freeze:** docs/02 Open Decisions must be EMPTY — everything
  Ratified or explicitly moved to post-1.0 with the owner's sign-off.
  Provisional items (S4, S5, S7, S13, S16, S17) get ratification rows.
  Stdlib signatures get a frozen reference page (docs/08-stdlib.md,
  generated from the sema builtin table by a script so it can't drift).

## Workstream 3 — Paper cuts & release engineering

- `lex --version`, `lex help` with real text, man-page-ish docs.
- Error-code web index: `docs/errors/E0xxx.md` generated from docs/04 +
  snapshots (each page: the error, a failing program, the fixed one).
- The language tour: docs/tour.md — 15 minutes, every feature, runnable
  (a test executes every tour snippet — the tour is examples too, I5).
- README rewrite around the showcase numbers; honest FAQ (vs Rust, vs
  Go, "where's async", "why semicolons").
- Release builds: prebuilt binaries (linux x86_64/aarch64, macOS,
  Windows) via CI; `lex upgrade` checks GitHub releases (just prints
  the URL — no self-modifying installs in v1).
- Versioning policy doc: post-1.0 syntax changes require a migration
  in `lex fmt` (the formatter is the upgrade tool).

## Exit criteria

- All three showcase tools build, pass goldens, and hit the ≤1.5×
  benchmark bar (recorded in showcase/BENCHMARKS.md with hardware).
- Fuzz soak: 24h local run, zero sema/rustc divergence.
- Open Decisions table empty; tour + error index generated and tested;
  binaries published for a tagged `v1.0.0`.
- The owner can hand a stranger the README and they ship a working tool
  the same afternoon — do one real external user test and record it.
